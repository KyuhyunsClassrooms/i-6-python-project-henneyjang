# 최종 보고서

## 1. 프로그램 주제와 목적

내 프로그램의 주제: 사용자 정의 함수와 2차원 리스트 기반의 JRPG 스타일 퀴즈 배틀 게임

이 프로그램을 만든 이유: 단순하고 지루한 퀴즈 프로그램에서 벗어나, 조건문과 반복문, 그리고 리스트 제어 문법을 활용하여 플레이어와 몬스터가 턴을 주고받으며 상호작용하는 몰입감 있는 교육용 게임을 개발하기 위함입니다.


---

## 2. 문제 분해 및 모델링

### 입력 
플레이어 이름, 난이도 선택(1 또는 2), 플레이어 행동 선택(1: 공격, 2: 포션), 퀴즈 정답 입력(1~3)


### 처리
난이도에 따른 초기 능력치(HP, ATK, 포션) 설정 및 퀴즈 데이터 무작위 셔플

플레이어 선택에 따른 대미지 계산 및 회복 연산

퀴즈 정답 여부 판정 및 정답 시 적에게 카운터 대미지 부여, 오답 시 적의 공격 대미지 계산

퀴즈 인덱스 증가 및 고갈 시 자동 재셔플


### 출력
매 턴 현재 캐릭터 상태(HP, 포션 개수) 시각화 출력, 퀴즈 질문 및 보기, 정답/오답 피드백 메시지, 최종 승패 결과

## 3. 사용한 2차원 리스트 설명

2차원 리스트 이름: quiz_pool

한 행의 의미: 하나의 독립된 퀴즈 문제와 관련된 모든 데이터 묶음

각 열의 의미:

| 열 번호 | 의미 | 실제 예시 |
|---:|---|---|
| 0번 열 |  |  |
| 1번 열 |  |  |
| 2번 열 |  |  |
| 3번 열 |  |  |
열 번호,의미,실제 예시
0번 열,퀴즈 질문 내용,"""대한민국의 수도는 어디일까요?"""
1번 열,1번 보기,"""1. 부산"""
2번 열,2번 보기,"""2. 서울"""
3번 열,3번 보기,"""3. 인천"""
4번 열,정답 번호 (문자열형),"""2"""
5번 열,정답 시 출력될 피드백,"""이건 맞춰야지요."""
6번 열,오답 시 출력될 피드백,"""잘못 누른거죠?"""
7번 열,정답 시 적에게 주는 대미지 배율 수치,3

이 2차원 리스트를 사용한 이유: 퀴즈의 질문, 보기, 정답뿐만 아니라 게임에 필요한 개별 연산 데이터(대미지 배율)와 고유 메시지까지 하나의 인덱스(quiz_index)로 묶어 유기적이고 체계적으로 관리하기 위함입니다.

## 5. 조건문과 반복문 활용

### 조건문 사용 부분

어떤 조건을 검사했는가? 난이도 설정 시 사용자의 입력을 판정할 때 (if mode == "2":)

플레이어가 공격할지 포션을 쓸지 행동을 분기할 때 (if player_choice == "1":)

퀴즈의 정답 여부를 판정할 때 (if user_choice == correct_answer:)


### 반복문 사용 부분

무엇을 반복 처리했는가? 플레이어와 몬스터 중 어느 한쪽이 쓰러질 때까지 전체 전투 흐름을 무한히 반복 처리할 때 (while player_hp > 0 and monster_hp > 0:)


### 조건문과 반복문이 함께 작동한 부분

while문 안에서 매 턴 퀴즈 인덱스가 증가할 때, 퀴즈 풀의 크기를 초과했는지 조건문으로 검사하여(if quiz_index >= len(quiz_pool):) 다시 처음으로 인덱스를 되돌리고 문제를 섞어주는 부분에서 함께 작동했습니다.

## 7. 테스트와 오류 수정

### 테스트 1: 정상 사례

입력: 난이도 1, 플레이어 행동 1, 퀴즈 정답 2 (정답 입력)

예상 결과: 플레이어가 적을 공격해 대미지를 입히고, 이어진 적 턴에서 퀴즈를 맞혀 카운터 대미지를 주고 체력을 방어함.

실제 결과: 예상대로 대미지가 차감되고 정답 메시지가 정상 출력됨.


### 테스트 2: 예외 또는 경계 사례

입력: 퀴즈를 계속 맞혀서 퀴즈 풀의 마지막 문제(15번째 문제)까지 도달한 상황

예상 결과: 인덱스 에러 없이 15번째 문제가 풀린 후, 바로 "문제를 새로 섞어 출제합니다"라는 안내와 함께 퀴즈 풀이 다시 섞여야 함.

실제 결과: 15번째 문제 출제 후 안전하게 리셋 및 셔플 처리됨.


### 발견한 오류와 수정한 방법

SyntaxError (문법 오류): 새로 추가한 퀴즈 데이터 중 둥근 따옴표(”)가 섞여 있거나 따옴표가 닫히지 않아 에러가 발생했으나, 곧은 따옴표(")로 교정하여 해결했습니다.

IndexError (인덱스 범위 초과): 퀴즈 리셋 검사문이 위에 있어서 마지막 문제를 푸는 순간 범위를 벗어나 터지는 버그가 있었으나, quiz_index += 1이 실행되자마자 그 즉시 인덱스를 검사하고 리셋하도록 순서를 교정하여 해결했습니다.

데이터 불일치 및 변수 범위 누락: 퀴즈 정답 시 몬스터에게 대미지를 주기 위해 함수에 기존에는 플레이어 턴 함수에서만 매개변수로 쓰던 monster_hp를 매개변수와 리턴 값으로 주고받도록 연동 구조를 수정했습니다.

## 8. Gemini 활용 내용 요약

Gemini에게 도움받은 부분: 코드를 기능별로 쪼개는 과정에서 생기는 변수 스코프(return과 매개변수)의 복잡한 연결 메커니즘을 깔끔하게 정리해 주었고, 실행 순서에 따른 IndexError와 마지막 데이터의 따옴표 오타(SyntaxError)를 정확히 짚어내어 디버깅해 주었습니다.


Gemini가 제안했지만 내가 수정하거나 사용하지 않은 부분: 잘못된 입력 시 입력 검증용 while 루프를 내부에 한 번 더 돌려서 올바른 값을 넣을 때까지 무한 반복 입력받는 구조를 제안했으나, 저는 가독성과 단순한 패널티 부여 기획을 위해 잘못 입력 시 턴이 넘어가게 둔 기존 코드를 그대로 유지했습니다.


최종 코드에서 내가 직접 판단하거나 수정한 부분: 광교고등학교 관련 상식 퀴즈, 6반 과목 관련 퀴즈 등 학교 생활과 관련된 개성 있는 오답 피드백 텍스트와 개별 퀴즈 난이도 배율 수치(60, 30, 10 등)는 제가 직접 기획하고 수정하여 채워 넣었습니다.

## 9. 한계와 개선 방안

현재 프로그램의 한계: 텍스트 기반 UI의 한계로 인해 전투의 생동감이 떨어지며, 플레이어가 잘못된 입력을 했을 때 올바른 값을 입력할 때까지 잡아두지 못하고 턴을 무조건 날려버리는 아쉬움이 있습니다.


다시 개선한다면 추가하고 싶은 기능:
플레이어가 공격 시 치명타(크리티컬)가 터지는 확률 시스템을 도입하고, 잘못된 입력이 들어왔을 때 예외 처리를 통해 "다시 입력하세요"라고 붙잡아두는 방어적 코드 검증 기능을 추가하고 싶습니다.
